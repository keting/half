# HALF Newcomer Path

[English](./newcomer-path.md) | [简体中文](./newcomer-path.zh-CN.md)

This guide is for contributors who are new to GitHub, open source
collaboration, or HALF. You do not need to start with code. Reproducing issues,
improving docs, updating screenshots, and starting research-oriented
Discussions are all useful contributions.

## Level 1: Understand The Project In 15 Minutes

- Read [`README.md`](../README.md).
- Browse the product screenshots and minimal-loop GIF in the README.
- Scan [`ROADMAP.md`](../ROADMAP.md).
- Browse Issues and Discussions to see existing topics and collaboration style.

After this level, you should understand:

- What HALF is and is not.
- Why HALF uses a human-in-the-loop model instead of executing agents directly.
- The main directions in the current roadmap.

## Level 2: Run The Project In Half A Day

- Follow [`docs/quickstart.md`](./quickstart.md) to start the local environment.
- Log in to the Demo Project.
- Explore the Plan DAG, task board, task details, and handoff prompts.
- Read [`docs/project-structure.md`](./project-structure.md) to learn the code layout.
- Record questions, documentation gaps, confusing interactions, or errors.

Those notes can become contributions: documentation improvements, bug reports,
UI/UX feedback, FAQ entries, or Discussions.

## Level 3: Make Your First Contribution

Good first contributions include:

- Fixing typos or unclear documentation.
- Adding missing Quick Start details, screenshots, or troubleshooting notes.
- Reproducing an open bug and adding steps, environment details, logs, or screenshots.
- Verifying whether a PR actually fixes its related Issue.
- Picking a small task labeled `good first issue`.

Before opening your first PR, check that:

- The change is small.
- The PR explains why the change is needed, what changed, and how it was verified.
- Docs, screenshots, or tests are directly related to this change.
- The diff does not contain secrets, private URLs, access tokens, or local paths.

## After Your First PR

After your first PR, you can try:

- Claiming engineering tasks labeled `status:ready`.
- Starting research ideas or roadmap proposals in Discussions.
- Contributing workflow templates, handoff prompts, or plan DAG cases.
- Recording agent collaboration failure modes, human intervention points, or
  benchmark / evaluation cases.
- Helping with Issue triage, PR review, and milestone discussion.

If you are not sure whether an idea should be an Issue, Discussion, or PR, read
the decision rules in [`CONTRIBUTING.md`](../CONTRIBUTING.md).
