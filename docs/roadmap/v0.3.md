# HALF v0.3 执行计划

本文档定义近期 v0.3 执行范围。它比根目录 `ROADMAP.md` 更具体，但仍是规划文档而非发布保证。

## 目标

v0.3 重点是将 HALF 的任务协调流程转变为更结构化、可复用、可追溯的 AI 编码工作流层。

任务卡片是工作流预设和轻量化 Handoff 的共享基础。移动端任务通知、GitHub URL 验证和管理员公共 Agent 在各自边界被接受后可以独立推进。

## 已承诺范围

### 任务卡片

将任务输入输出标准化为结构化任务卡片。

v0.3 应定义工作流模板、handoff、提示生成和后续实验记录所需的最小任务卡片字段。设计应扩展现有任务模型，而非引入并行任务系统。

预期产出：

- 任务卡片 schema 决策记录
- 后端/API 支持所选字段
- 适当的 UI 展示和编辑
- 与现有任务生命周期行为兼容

### 工作流模板预设

使用现有的 `ProcessTemplate` 模型作为工作流模板载体。v0.3 不得引入第二个工作流模板模型或工作流引擎。

初始预设：

- 单 Agent
- 开发 + 测试
- 开发 + 审查
- 开发 + 测试 + 审查
- 需求到测试（作为工作流模板示例）

预期产出：

- 基于现有流程模板机制的预设模板
- 可复用工作流模式的示例目录
- 仅在任务卡片或 handoff 需要时的最小字段增强

非目标：

- 分支工作流逻辑
- 循环
- 动态路由
- 工作流特定的 handoff 策略

### 轻量化 Handoff

为任务节点定义轻量化 handoff 格式。

v0.3 应支持：

- `handoff.json` schema
- 从任务结果目录读写 handoff 数据
- 任务详情视图中只读展示
- 后继提示中的定字段 handoff 包含

定字段包含：

- `summary`
- `modified_files`
- `artifact_references`
- `test_status`
- `known_risks`

包含规则：

- 仅包含直接前驱任务的 handoff
- 多个前驱按依赖顺序列出
- 缺失或无效的 handoff 数据不阻塞派发
- 长内容会被截断，但保留制品引用

非目标：

- 用户可编辑的包含预览
- 字段级包含配置
- 工作流特定的包含策略
- 自动摘要、合并、去重或冲突检测
- 累积外部化状态集成
- 通用上下文组装引擎

### 移动端任务通知

添加移动端友好的任务状态通知支持。

飞书、企业微信、通用 webhook 或其他移动端可达通道可作为初始实现路径评估。v0.3 应选择单一最小通道，而非构建多通道通知平台。

初始事件：

- 任务完成
- 任务失败或需要关注
- 任务需要人工输入

初始负载：

- 项目名称
- 任务名称或任务代码
- 状态
- 分配的 agent
- 可用时的简短摘要
- 可用时的 HALF 页面链接

非目标：

- 邮件通知
- 通用通知管理平台
- 交互式审批按钮
- 来自通知通道的入站回调
- 通知投递失败时阻塞任务状态流转

### 管理员公共 Agent

支持管理员管理 Agent 的简单公共 Agent 目录。

v0.3 的预期方向是窄范围：

- 管理员管理的公共 Agent 对所有用户可见
- 普通用户可以在允许的地方使用公共 Agent
- 普通用户不能编辑管理员管理的公共 Agent
- 普通用户的私有 Agent 保持私有

ADR 0004 中的访问控制语义必须在实现前被接受。此项应保持简单，不得扩展为通用资源共享或多租户治理。

### GitHub 仓库 URL 验证

添加仓库 URL 验证作为独立缺陷修复。

预期行为：

- 常见错误的客户端格式验证
- 后端验证，优雅处理不可达的私有仓库
- 清晰区分格式错误 URL 与连接或凭证问题的错误信息

## 纯研究项目

### Agent 限制/重置时间检测

v0.3 可研究是否可以在不跨越合规边界的情况下获取 Agent 限制和重置时间数据。

预期产出：

- `docs/research/` 下的研究笔记
- 实际可行时覆盖 Claude Code、Codex、Copilot、GLM 和 Kimi
- 清楚分类官方支持 API、人工输入、日志、UI 提示和不支持/私有方法

此项工作不得将自动检测集成到 v0.3 产品中。

### 累积外部化状态

v0.3 可研究累积外部化状态作为未来能力。

预期产出：

- `docs/research/` 下的研究笔记或 ADR 草案
- 候选字段和示例记录
- 与任务卡片、handoff、实验运行日志和失败记录的关系

此项工作不得将累积外部化状态集成到 v0.3 运行时。

## 依赖

推荐执行顺序：

1. 在相关实现开始前将 ADR 0001、0002 和 0004 从 proposed 提升为 accepted。
2. 独立实现 GitHub URL 验证和移动端任务通知。
3. 实现任务卡片存储、API 和 UI 界面。
4. 使用现有 `ProcessTemplate` 模型添加流程模板预设。
5. 实现轻量化 handoff 读写、展示和直接前驱 handoff 在后继提示中的包含。
6. 在 v0.3  happy path 可演示后添加 README 演示资源。
7. 并行完成纯研究笔记。

## 非目标

v0.3 不交付：

- 订阅制 coding agent 的自动调用
- 本地 Agent 桥接
- API Agent 连接器
- 配额感知执行
- 受控自动化
- 作为产品功能的策略或人工审查门
- 作为产品功能的企业案例记录
- 累积外部化状态集成
- 通用工作流引擎
- 通用上下文组装引擎

## 验收说明

以下情况视为 v0.3 成功：

- 任务卡片为模板和 handoff 提供稳定结构化输入
- 初始工作流预设可在不使用第二个模板模型的情况下使用
- handoff 数据在下游提示中通过定字段直接前驱包含可见且有用
- 移动端任务通知提供有用的外部状态感知而不影响任务状态流转
- 公共 Agent 可见性保持窄范围且易于理解
- 纯研究项目产出可决策文档而不扩展 v0.3 实现范围
