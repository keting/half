# 人机协同多智能体任务管理平台（Human-AI Loop Framework，HALF）

## 当前阶段目标与非目标

**本文档范围说明：** 本版 PRD 以 **v0.x early open-source MVP** 为范围基准，用于指导实现一个可跑通核心闭环的原型系统。

**当前阶段主目标：** 面向小团队和单租户自托管场景的人机协同多智能体任务管理原型，优先保证可用、够稳、能在真实 Agent 环境中跑通核心流程闭环。

**次目标：** 为后续版本积累可观测性、流程沉淀与数据结构化基础。

**暂不追求：** 通用平台化能力，包括低门槛上手、复杂运维能力、广泛 Agent 适配等，保留为后续验证方向。

**MVP 成功标准：**

- 至少一个真实项目使用本系统完成从项目创建到执行汇总的全流程闭环。
- 参与执行的命令行类 Agent ≥ 2 个。
- 负责人能够在页面内完成计划生成、计划调整、任务派发、状态查看和异常处理。
- 系统接入真实 Agent，非纯 mock 演示。

## 背景与动机

AI Coding / Coding Agent 的实践近段时间以来快速推进，同时也面临以下问题：

（1）**多 Agent 协同的操作负担已成为实际瓶颈。** 使用单一 Agent 来解决研究和实践问题已在有效性和效率上显现出局限，多 Agent 协同正成为常态。然而，即便是最常见、最基本的协同场景——例如使用 Claude Code 和 Codex 完成一个研发项目的需求评审——也往往涉及七八个步骤的穿插调度：每一步都需要手工复制 Prompt 发送给对应的 Agent，反复查看任务是否执行完成，再决定进入下一个流程。步骤一多就容易遗漏或搞乱顺序，影响效率。当场景更加复杂、涉及更多 Agent 时，这种人工协调的负担会愈发突出。

（2）**多 Agent 协同需求与资源约束的矛盾。** 在真实项目中，不同 Agent 的能力边界、可用性和使用成本并不一致；与此同时，不同 Agent 在特定任务上各有优势，多个 Agent 协同工作有望实现能力互补、提升整体交付效率。

（3）**缺乏共性平台支撑工程实践。** 理想情况下，团队应有一个 AI Coding / Coding Agent 领域的共性平台，为研究和工程实践提供基础设施支撑：系统为 Context 管理、多 Agent 协同、人机协同等方向提供落地场景和集成环境，实践反馈再反过来提升系统的功能与技术含量，逐步形成良性互动与技术积累。

本系统正是为了应对以上问题而构思。MVP 阶段聚焦于先跑通一个真实、合规、可演示的闭环，后续再逐步扩展到更完整的平台能力。

## 系统定位

本系统是一个纯人工触发、合规安全、无自动调用、无接口逆向的人机协同多智能体任务管理平台。MVP 重点面向 **本地 CLI 版本的命令行类 Agent** 设计，优先支持 Claude Code 与 Codex，在合规前提下实现任务流程编排、人工分发、信息共享、状态跟踪、结果归档与进度可视化。

MVP 阶段不支持网页对话类 Agent 作为标准执行角色；相关支持保留到后续版本探索。

### MVP 范围说明

**MVP 包含：**

- 登录与注册界面，支持用户名 + 密码注册和登录（密码要求包含大小写字母和数字，最少8位）
- 项目创建页面，填写项目目标、选择参与 Agent、填写 Git 仓库地址；若某个 Agent 配置了多个模型，则在选择卡片中展示全部模型名称
- 项目列表页面，展示项目名称、状态、参与 Agent 数量、创建时间
- 项目详情页，以“下一步应该做什么”为核心提示
- 工作计划生成支持两种来源：由 Prompt 生成流程，或使用流程模版直接生成任务
- Prompt 规划路径中，系统推荐规划 Agent，Prompt 一键复制，支持可选查询模型用量，并在 Plan 页面选择规划模式（均衡模式 / 效果优先 / 性价比高 / 速度优先）
- 流程模版管理页面，支持创建、查看、编辑、删除流程模版，使用人机协同 Prompt 生成模版 JSON，预览 DAG，为每个抽象角色槽位维护可选角色说明，并可声明模版必需输入信息；在项目 Plan 页面将模版角色槽位映射到项目 Agent、填写模版必需输入后直接生成任务
- 工作计划查看与调整，系统自动轮询协作目录中的唯一规划文件并回填页面，负责人可手工修订后确认
- 任务执行页，按 DAG 展示待执行任务并支持 Prompt 一键复制
- 任务状态跟踪，基于 Git 轮询、任务码路径 + 内容双写检测
- 基础异常处理，支持重新派发、手动标记完成、标记放弃
- 执行汇总页面，展示任务状态与产出文件链接
- Agent 总览页面，展示所有智能体的名称、类型、模型列表、各模型对应的能力描述、可用状态、短期重置剩余时间（倒计时格式，悬浮显示具体时间）、短期重置间隔、长期重置剩余时间（倒计时格式，悬浮显示具体时间）、长期重置间隔

**MVP 不包含：**

- 网页对话类 Agent 支持
- 工作计划评审步骤（归入 V1）
- IM 通知集成
- 智能分派算法
- 服务器管理页面
- 结构化执行报告文档生成

### 设计选型说明

- **为什么项目协同数据使用 Git？** 所有参与的命令行类 Agent（如 Claude Code、Codex）天然具备 Git 读写能力，以 Git 仓库承载任务输出和协同结果可实现零额外集成成本。同时 Git 的版本历史天然满足可追溯、可审计的合规需求，且写在 Git 上的内容既可被 Agent 读取，也可被人类成员读取。
- **为什么系统状态不完全存 Git？** Git 适合承载任务输出与协同记录，但不适合承担登录态、项目元数据、任务状态主记录、页面交互状态等系统内部状态管理。MVP 采用 **SQLite 存系统状态，Git 存项目协同数据** 的双层方案，以降低实现复杂度并提高可操作性。Git 轮询结果同步更新 SQLite 中的任务状态主记录。
- **为什么用轮询而不是 webhook / 事件驱动？** 合规约束下系统不能调用 Agent 平台的非公开接口，也不能在 Agent 侧植入回调钩子。Git 轮询是在该约束下最可行的状态检测方式。MVP 采用可配置轮询间隔（全局默认 15-30 秒，支持项目级覆盖），并提供手动刷新按钮。系统支持配置轮询延迟启动时间（全局默认 0 秒，支持项目级覆盖）。
- **为什么任务结果改为固定任务目录 + `result.json` 完成哨兵？** 真实执行中，`expected_output` 往往会混入自然语言描述、多文件说明或括号注释，继续依赖它做轮询定位会非常脆弱。MVP 当前将任务完成契约收敛为：每个任务固定写入 `{collaboration_dir}/{task_code}/`，所有真实产物先写完，再最后原子提交 `result.json` 作为完成哨兵；可选 `usage.json` 与其放在同目录。这样可同时覆盖单文件与多文件任务，并显著降低轮询误判与漏判。

## 角色说明

系统本身作为基础设施，负责任务编排、状态轮询、信息归档与页面展示，不作为独立参与者介入执行过程。

### 项目负责人（Owner）

项目的创建者与整体驱动者。负责创建项目、选择参与 Agent、生成和调整计划、派发任务，并对项目的进度和结果负全责。

### Agent（智能体）成员

参与项目执行的 AI 工具。MVP 中仅包含命令行类 Agent，本地 CLI 版本运行，可独立接收 Prompt、完成任务并按规范将结果写入 Git 仓库。

### 人类成员

MVP 当前提供基础登录能力和项目级操作隔离。所有登录用户仍以“项目负责人”身份操作自己创建的项目；系统暂不提供协作者、多角色共享项目或多人共同编辑同一项目的流程。所有人工操作角色（复制 Prompt、发送给 Agent、确认异常处理等）仍由当前项目负责人承担。

### 权限说明

MVP 提供用户名 + 密码注册与登录，密码要求包含大小写字母和数字、最少8位。系统当前区分两类用户：

- **管理员（admin）**
  - 可访问和修改系统级配置
  - 可访问智能体设置页面
  - 可访问项目参数设置页面
  - 可访问用户管理页面，查看所有用户并修改用户角色/状态
- **普通用户（user）**
  - 可管理自己创建的智能体、项目及其派生数据
  - 不可访问系统级设置与用户管理

系统按当前登录用户对 Agent / Project / Plan / Task / Polling 等业务资源做 owner 级隔离：用户只能看到并操作自己创建的智能体、项目及其派生数据；跨用户访问应返回资源不存在或参数非法。管理员当前**不接管其他用户的项目和智能体**，其管理范围聚焦于用户管理和系统级设置。

用户状态分为：

- `active`：正常，可登录并使用系统
- `frozen`：冻结，禁止登录；若用户已持有 token，则后续请求也会立即被拒绝

系统必须至少保留一个激活状态的管理员。管理员不能冻结自己，也不能修改自己的角色。

对于删除智能体的引用校验，系统仅检查**当前登录用户自己的项目与任务**是否仍引用该智能体；其他用户项目中的历史脏引用不应阻止当前用户删除自己的智能体，但当前用户自己项目/任务中的真实引用仍必须返回删除失败。

## 协作流程

项目负责人创建项目后，选择参与的命令行类 Agent，并根据执行计划将任务分配给对应 Agent 执行。MVP 中所有任务分发均由项目负责人手工完成。

任务分配方式在 MVP 中收敛为一种主模式：

1. **直接分配给 Agent：** 系统生成 Prompt，负责人手工复制并发送给对应 Agent 执行。

后续版本可扩展人工成员协作、指定代理人代发 Prompt、纯人工任务等复杂模式，但不作为本期范围。

## 核心使用流程

### 环境前提

用户拥有一个或多个独立运行环境，这些环境可以是独立服务器、团队成员各自的电脑，或两者混合。

MVP 重点支持多台机器通过 Git 远程仓库同步，每台机器上运行一个或多个不同种类的本地 CLI Agent。

### 登录系统

用户首先通过用户名和密码登录 Web 管理平台。首次使用的用户是否可以自助注册，由部署配置 `HALF_ALLOW_REGISTER` 决定：

- 当 `HALF_ALLOW_REGISTER=true` 时，登录页展示注册入口；用户可提供用户名和满足强度要求的密码（包含大小写字母和数字，最少8位）完成注册，注册成功后自动登录并进入系统。
- 当 `HALF_ALLOW_REGISTER=false` 时，登录页不再展示注册入口，并明确提示“当前环境未开放自助注册”；后端同时拒绝注册请求。

该登录机制用于支持基础身份识别与访问控制。

新注册用户默认状态为正常（`active`）。

已登录用户可在左侧导航栏底部通过“修改密码”入口打开密码修改弹窗。弹窗包含“当前密码 / 新密码 / 确认新密码”三个字段，并遵循以下规则：

- 当前密码必须校验通过，否则提示“当前密码错误”
- 新密码必须满足现有强度规则（包含大小写字母和数字，最少8位）
- 新密码不能与当前密码相同
- 确认新密码必须与新密码一致
- 修改成功后关闭弹窗并提示成功

错误场景下，系统应保留当前登录态，不应因为业务校验失败把用户误判为登录失效并跳回登录页。

对于历史遗留数据，如果旧版本项目或智能体记录缺少创建者（`created_by IS NULL`），系统需要在升级时完成兼容处理，避免 owner 校验导致旧数据全部不可见。

### 时间展示规则

项目创建/更新时间、计划派发/检测/创建时间、任务派发/完成/创建时间、任务事件时间、用户注册/最后登录时间等运行事件时间，后端统一按 UTC 存储和传输；API 返回时必须带明确 UTC 时区标记（如 `Z` 或 `+00:00`），避免浏览器把无时区字符串误当成本地时间解析。

前端展示上述运行事件时间时，按用户浏览器本地时区格式化显示，不额外追加 `UTC+8`、`Asia/Shanghai` 等时区标识。

Agent 短期/长期重置时间的时区统一改造作为后续独立事项处理；在该事项实施前，Agent 重置时间仍按现有规则处理。

### 创建项目

项目负责人在 Web 管理平台创建项目，填写项目名称、任务目标、参与 Agent，并填写项目对应的 Git 仓库地址。MVP 假设每个项目对应一个独立 Git 仓库。

项目创建/编辑时，已处于"不可用"状态的 Agent 仍展示在列表中，但不能被**新增**为项目参与者；编辑已有项目时，若某个不可用 Agent 已经存在于该项目的当前参与集合中，则允许继续保留并随项目一起保存。短期重置待恢复和长期重置待恢复不属于本条限制范围。

**规划模式**：规划模式不再出现在项目创建/编辑页，而是在 Plan 页面选择“由 Prompt 生成流程”时展示和保存。该字段持久化为 `planning_mode`，用于生成规划 Prompt 时指导任务拆分、Agent 分配、模型选择和并发/评审策略。可选值包括：`balanced`（均衡模式）、`quality`（效果优先）、`cost_effective`（性价比高）、`speed`（速度优先）。默认值为 `balanced`；负责人点击“生成 Prompt”时，系统会同时保存任务介绍和本次选择的规划模式。

**协作目录**：每个项目在 Git 仓库内有一个独立的协作目录（plan 文件和任务输出文件都写入该目录）。该字段可在创建/编辑表单中显式填写；若用户留空，系统自动设为 `outputs/proj-<项目id>-<随机串>`，其中 `<项目id>` 是系统创建项目时分配的唯一自增 ID，`<随机串>` 为系统生成的短随机后缀。这一默认值可进一步降低目录碰撞概率，避免多个项目共用同一目录引起的写入混乱。

### 配置 Git 仓库

MVP 阶段默认由项目负责人手动准备 Git 仓库，并在创建项目时填入仓库地址。系统负责基于该地址执行 clone、pull、读取和轮询。自动创建仓库属于增强项，不作为本期必需能力。

> 轮询读取策略：生产场景优先读取后端维护的仓库副本；若部署时后端与用户工作区位于同一台机器，且挂载了共享工作区目录，则系统可在仓库副本未命中文件时，按同一 Git remote 校验后回退读取共享工作区中的同路径文件。这样即使结果文件尚未 push 到远端，只要已经写入当前工作区，也能被“手动刷新”或后台轮询检测到。
>
> 对于多机器协作场景，轮询不能只依赖本地工作树是否能 `git pull --ff-only` 成功。系统应优先 `git fetch origin`，必要时直接读取远端跟踪分支（如 `origin/main`）上的目标文件快照，从而在本地缓存仓库已分叉时，仍能识别其他机器后续 push 的任务结果。
>
> 若 `git fetch origin` 已成功、但后续 `git pull --ff-only` 因多 Agent 并行推送导致本地缓存仓库无法 fast-forward，系统应将该情况作为后端诊断 warning 记录到日志中，继续使用可达的远端快照完成检测；该 warning 不写入计划或任务的 `last_error`，也不作为用户需要处理的“最近错误”展示。

### 进入项目详情页

创建完成后进入项目详情页。页面最醒目的核心 UI 元素是 **“下一步应该做什么”** 的提示，始终引导负责人执行当前最关键操作。

### 第一步：生成工作计划

进入 Plan 页面后，负责人首先选择流程来源。流程来源选择应表现为一个紧凑的分段控件，用于表达“顶层开关”而不是普通卡片选项；顺序为左侧 **使用流程模版生成流程**、右侧 **由 Prompt 生成流程**。页面初始默认优先进入模版路径；当真实项目数据和流程模版列表加载完成后，如果不存在任何项目可用模版，则自动切回 Prompt 路径。这里“可用模版”仅按项目已选 Agent 数量判断：`project.agent_ids.length >= template.agent_count`。系统应在同一浏览器内按用户和项目记忆负责人上一次手动选择的流程来源；再次进入同一项目 Plan 页时，合法记忆值优先于默认值。无记忆、记忆值非法，或记忆为模版但当前项目没有可用模版时，必须回退到上述默认逻辑。该自动校正只发生在项目进入/切换时，负责人手动切换后不得被后续刷新、轮询刷新或页面重新聚焦覆盖；同一 SPA 会话内从项目 A 直接切到项目 B 时，也必须重新读取项目 B 自己的记忆或默认值，避免不同项目互相污染。

- **使用流程模版生成流程**：负责人填写或补充本次任务介绍后，从已有流程模版中选择一个模版，将模版中的抽象角色槽位（如 `agent-1`、`agent-2`）映射到本项目内的具体 Agent。槽位映射必须完整且不能重复；模版所需角色数量不得超过项目可用 Agent 数量。映射表单应在每个槽位下方展示该槽位的角色说明（承担职责和适合绑定的 Agent 类型），未填写时展示“暂无说明”。若模版声明了必需输入信息，Plan 页应在角色映射区域下方展示“模版所需信息”结构化表单，并在必填字段为空时禁用“下一步”。确认应用前，系统必须校验任务介绍非空，并先将任务介绍保存到项目 `goal`、将当前模版声明字段对应的填写值保存到项目 `template_inputs`；该路径不保存或修改 `planning_mode`。确认应用后，系统直接创建最终计划和任务并进入任务执行页，不再生成规划 Prompt、不启动规划轮询。
- **由 Prompt 生成流程**：负责人填写或补充本次规划的任务介绍，包括目标、约束、交付物和验收标准。页面不再提供单独的“确认任务介绍”按钮；任务介绍和规划模式会在点击“生成 Prompt”时自动保存。随后负责人需要勾选本次参与规划的 Agent 列表，系统在生成 Prompt 时会把这些 Agent 的数量和特性带入规划上下文，用于约束后续子任务拆分。若某个 Agent 配置了多个模型，则页面允许负责人为该 Agent 在本项目中指定一个模型；若仅勾选了 Agent 而未指定模型，则系统会根据任务目标、项目规划模式和模型能力描述自动选择最适合的模型。

Plan 页面应提供以下能力：

- 流程来源选择：使用流程模版生成流程 / 由 Prompt 生成流程；默认优先模版路径，无可用模版时自动切回 Prompt；用户手动选择后按当前用户和项目记忆，非法记忆或不可用模版记忆回退默认
- 任务介绍输入框及自动保存辅助说明；Prompt 路径在点击“生成 Prompt”时保存任务介绍和规划模式，模版路径在点击“下一步”应用模版前保存任务介绍和模版必需输入值
- Prompt 路径下的规划模式选择器
- Prompt 路径下的参与规划 Agent 勾选区
- Prompt 路径下已选 Agent 的模型选择器（每个 Agent 最多指定一个模型，可留空自动选择；用户手动指定模型优先级高于模式建议）
- 模版路径下的流程模版列表、模版说明、所需角色数、抽象角色槽位、槽位角色说明、槽位到项目 Agent 的映射表单，以及按所选模版 `required_inputs` 渲染的“模版所需信息”表单
- Agent 勾选卡片第一行按“名称 + 空格 + agent type”展示，避免名称与类型连写
- “是否查询模型用量”勾选框
- “生成 Prompt”按钮
- “拷贝 Prompt”按钮
- 当前状态灯与计时器

负责人点击“生成 Prompt”后，系统只生成本次 Prompt，不启动轮询；Prompt 内容会根据是否勾选“是否查询模型用量”而变化，并写明本次各 Agent 最终采用的模型。若用户未手工指定模型，则 Prompt 中应使用系统自动选择后的结果；自动选择应将项目规划模式作为额外参考，但在当前没有结构化成本/速度字段时仅能基于模型能力描述做软匹配。同一项目在尚未点击“拷贝 Prompt”派发前，重复点击“生成 Prompt”应复用同一条未派发候选计划记录和同一个 `plan-<规划记录ID>.json` 输出路径，只更新 Prompt 内容、参与 Agent、模型选择等参数，不应静默创建新 plan_id 导致负责人已交给外部 Agent 的旧 Prompt 路径失效。只有该计划已经派发、完成或结束后，再次生成 Prompt 才表示开启新一轮规划候选。

规划 Prompt 必须包含与当前 `planning_mode` 匹配的策略段落，并将其作为任务拆分、Agent/模型分配、并发度和评审策略的核心约束，而不是只作为风格说明：`balanced` 综合能力匹配并避免不必要重复；`quality` 可将关键目标拆成多个并行候选 task 再增加评审/合并 task，但不得要求单个 task 绑定多个 assignee；`cost_effective` 在能力满足要求时优先低成本或节省用量的模型，且用户手动指定模型优先；`speed` 尽量减少串行依赖、提升可并发度并缩短关键路径。

规划 Prompt 中的参与 Agent 列表必须保留项目级“同服务器：是/否”标记。系统还应在参与 Agent 说明之后、输出要求之前加入全局“同机分配引导”文案，用于指导规划 Agent 将依赖本地部署环境访问的任务优先分配给同服务器 Agent；不依赖本地环境的任务可分配给任意合适 Agent。该引导应始终加入，即使本轮参与 Agent 中没有同服务器 Agent，也需要提示规划 Agent 在任务描述中标注同机执行风险。

负责人点击“拷贝 Prompt”后，系统会：

1. 将 Prompt 复制到剪贴板
2. 将当前未派发候选计划推进为“计划生成中”；若当前计划已经派发、完成或结束，再按重新派发语义创建新一轮候选计划
3. 启动后台轮询，到项目 Git 仓库协作目录中检查本轮规划对应的唯一结果文件，例如 `plan-<规划记录ID>.json`

> 协作目录路径约定：项目创建/编辑时填写的"协作目录"必须是仓库根相对路径（如 `v2`、`outputs/v2`），不得以 `/` 开头。系统会在保存时自动 strip 前后斜杠并规范化为仓库根相对形式。

> 轮询健壮性要求：若后端本地缓存仓库 `git pull --ff-only` 因分叉等原因失败，但远端快照仍可读取，系统不应直接放弃本轮状态检测，而应继续使用现有 checkout、可用的共享工作区回退源以及远端跟踪分支快照完成本次文件检查，并将该情况记录到后端日志中，避免页面长期停留在“轮询中”。该类可降级 warning 不写入 `last_error`；若超过计划生成超时时间仍未检测到合法计划 JSON，仍按正常计划超时进入“需人工处理”。

若超过设定时间（默认 30 分钟）仍未检测到合法的计划 JSON，页面将该次计划生成标记为“需人工处理”，提示负责人检查 Prompt 是否已发送、Agent 是否已按约定回写文件。

当前状态使用三种灯色：

- 黄灯：未启动，或上一轮轮询已结束但未查询到结果
- 红灯：用户已点击“拷贝 Prompt”，系统正在轮询结果
- 绿灯：已查询到合法规划结果

当状态为红灯时，页面需实时显示从本次页面中点击“拷贝 Prompt”开始累计的耗时计时器，计时必须从 `00:00:00` 开始；若只是重新进入页面而未在当前会话中重新点击“拷贝 Prompt”，则不直接延续显示历史计时。

若用户重复点击“拷贝 Prompt”，系统行为如下：

- 若当前轮询仍在运行，则不重复启动新轮询
- 若当前轮询已结束（已查询到结果，或已超时结束），则启动新一轮轮询

### 工作计划输出格式

MVP 规定工作计划必须输出为 **结构化 JSON**，由系统作为唯一解析输入，并渲染为可读 DAG 视图。负责人可在页面上调整任务节点、依赖关系与执行对象。

结构化计划至少应包含以下字段：

- `plan_name`：计划名称
- `tasks`：任务列表
- `tasks[].task_code`：任务码
- `tasks[].task_name`：任务名称
- `tasks[].description`：任务描述
- `tasks[].assignee`：执行 Agent 标识
- `tasks[].depends_on`：前置任务码列表
- `tasks[].expected_output`：预期输出说明

PRD 要求输出必须为合法 JSON，可被程序稳定解析，不接受自由文本或 YAML 作为正式计划来源。

流程模版也使用同一类结构化 JSON，但 `assignee` 必须使用抽象角色槽位，格式为 `agent-N`（N 从 1 开始），不得直接绑定真实 Agent slug。模版保存时必须校验任务码唯一、必填字段非空、依赖只引用已存在任务、依赖图无环、`expected_output` 为仓库根相对路径。模版名称最终不能为空；创建或更新时若请求未显式传入名称，则从 JSON 的 `plan_name` 派生。模版描述允许为空；若请求未显式传入描述，则从 JSON 的 `description` 派生。流程模版还应独立保存用户在“输入描述”中填写的详细流程描述，用作后续再次生成模版编写 Prompt 的来源文本；该字段与适用场景描述、模版 JSON 和项目任务介绍互不混用。流程模版可额外声明 `required_inputs` 结构化字段列表，用于表达使用该模版生成流程前必须补充的信息；每项包含 `key`、`label`、`required`、`sensitive`。`key` 必须非空、符合 `[a-zA-Z_][a-zA-Z0-9_]*` 且在同一模版内唯一；`label` 必须非空；`required` 和 `sensitive` 必须为布尔值。`sensitive` 仅影响 Plan 页输入框以 password 样式渲染，不表示加密或脱敏。所有登录用户均可查看和使用流程模版，只有创建者和管理员可编辑或删除模版。删除模版不影响已经由该模版生成的项目任务。

流程模版允许在 JSON 顶层提供可选 `agent_roles` 数组，用于描述每个抽象角色槽位的职责和适合绑定的 Agent 类型，例如 `[{"slot": "agent-1", "description": "负责需求拆解，适合产品/架构类 Agent"}]`。页面预览 JSON 时应按 `tasks[].assignee` 自动抽取真实槽位数量，角色说明区只为当前存在的槽位生成输入框；用户不能在角色说明区独立增删槽位，如需增删角色必须修改任务中的 `agent-N` 后重新预览。JSON 中的 `agent_roles` 用于预填和同步角色说明：空输入可直接预填；若某个槽位的当前说明仍等于上一次 JSON 预填值，则再次预览时应随新的 JSON 说明更新；若用户已经在角色说明输入框中手工编辑过该槽位，则再次预览不得覆盖用户手工输入。保存时后端只持久化当前槽位对应的非空说明，过滤无效槽位、空字符串和已被删除的槽位；角色说明仅用于创建/编辑/详情和 Plan 映射辅助展示，不影响任务生成、槽位数量计算或 Agent 映射校验。

最小完整 JSON 示例：

```json
{
  "plan_name": "多 Agent 文献综述与实验方案设计",
  "tasks": [
    {
      "task_code": "TASK-001",
      "task_name": "检索相关文献",
      "description": "检索近三年与主题相关的核心论文并整理摘要",
      "assignee": "claude-code-01",
      "depends_on": [],
      "expected_output": "outputs/TASK-001/result.json"
    },
    {
      "task_code": "TASK-002",
      "task_name": "整理研究脉络",
      "description": "基于文献结果归纳研究问题、方法路线和空白点",
      "assignee": "codex-01",
      "depends_on": ["TASK-001"],
      "expected_output": "outputs/TASK-002/result.json"
    },
    {
      "task_code": "TASK-003",
      "task_name": "形成实验方案草案",
      "description": "结合研究脉络输出实验方案、指标和风险点",
      "assignee": "claude-code-01",
      "depends_on": ["TASK-002"],
      "expected_output": "outputs/TASK-003/result.json"
    }
  ]
}
```

### 确定最终计划

系统轮询到合法的规划结果文件后，不再在 Plan 页面展示文字版、JSON 版或图表版结果，也不再提供“确认规划”按钮。系统应直接将该结果定稿为最终计划，并自动跳转到任务执行页。

当负责人使用流程模版时，系统在应用模版时完成定稿：先保存当前任务介绍到项目 `goal`，并将当前模版声明的必需输入值保存到项目 `template_inputs`，再将模版中的 `agent-N` 槽位替换为映射后的项目 Agent，创建一条已完成的候选计划并立即转为最终计划，同时生成任务。该路径不依赖 Git 规划文件，因此 `source_path` 记录为 `template:<template_id>`，轮询服务应跳过这类来源，避免误读仓库路径。应用模版前，系统会清理同项目下旧的未选中候选计划，保留已定稿的最终计划语义。任务介绍和模版必需输入值不写入流程模版 JSON 或 task.description；任务介绍作为项目 `goal` 提供执行背景，模版必需输入值由执行 Prompt 根据任务来源模版统一注入。

确认后的计划作为后续任务执行的唯一依据。Plan 页面不再承担人工评审结果视图；负责人若需要调整任务内容，应在任务执行页中针对当前可执行任务直接修改右侧文本信息，系统自动保存。

系统优先按严格 JSON 解析规划结果；若 Agent 输出中仅存在常见的格式性问题（如字符串中的引号未转义），系统可进行有限度的自动修复后再尝试解析。

MVP 不包含独立的“工作计划评审”流程；若负责人需要参考多个 Agent 生成的计划，以最新一轮成功回写的合法结果为准。

### 执行阶段

项目详情页根据最终计划，按 DAG 顺序展示每个任务的分配对象、前序依赖、输出要求和操作指引。

对于 Agent 任务：

- 展示可一键复制的 Prompt
- 「复制 Prompt 并派发 / 重新派发」按钮必须保证 *复制* 与 *派发* 的原子性：剪贴板里的内容、按钮上的「已复制」提示、以及后端的派发记录三者必须同时成功或同时失败；任何一种复制失败（包括浏览器剪贴板权限被拒、user activation 失效、fallback 失败等）都必须显式提示用户并取消本次派发，绝不允许出现"显示已复制但剪贴板里其实是上一次任务 Prompt"的情况
- 默认只要求输出任务结果
- 可选勾选”输出剩余用量”，勾选后在 Prompt 中附加用量输出指令
- 若该任务已有用量上报数据，在任务详情中直接展示
- 每个任务需要明确显示指派给哪个 Agent 执行

对于存在前序依赖的任务：

- 只有当前序任务全部处于“已完成”或“已放弃”状态时，当前任务才允许操作
- 若任一前序任务尚未完成，则当前任务的“复制 Prompt”和“放弃任务”都不得点击
- HALF 页面不负责检查“前序任务输出文件是否已在仓库中可见”；该检查在页面侧并不可靠，也不应作为派发前提
- 若前序任务状态为“已放弃”，后继任务可继续执行，系统不再要求该前序任务必须存在输出文件
- 派发给 Agent 的 Prompt 会强制要求 Agent 在执行任务前先在本地仓库执行 `git pull`，并检查 Prompt 中列出的前序任务输出文件是否真实存在；若不存在，应停止执行并与项目负责人沟通
- 左侧 DAG 中，不可执行、当前可执行、执行中、执行异常、执行成功的任务节点需使用不同底色
- 当前可执行任务被选中后，右侧任务名称、任务描述、预期输出等文本允许直接修改，修改后自动保存

项目负责人负责将 Prompt 发送给对应 Agent，并等待回写结果。

### Prompt 模板草案

系统与 Agent 的核心交互接口为 Prompt。MVP 阶段推荐使用以下模板结构：

```text
你是项目 [{项目名称}] 的执行 Agent。

{如果项目 goal 非空：}
## 项目任务介绍
{项目 goal 去除首尾空白后的原文}

{如果当前 task 来源于流程模版且项目 template_inputs 中存在该模版声明字段的非空值：}
## 模版所需信息
- {字段 label}: {字段 value}

## 执行前置步骤（必须先做）
1. 在开始本任务前，必须先在项目仓库目录执行 `git pull`，确保拿到最新的远端状态，否则可能读不到前序任务输出。
2. 确认上述前序任务输出文件已经存在；若仍缺失，请等待或与项目负责人沟通，不要凭空创作前序内容。

## 任务信息
- 任务码：{TASK-XXXX}
- 任务名称：{任务名称}
- 任务描述：{任务描述}

## 前序任务输出
{前序任务的输出目录列表，如：outputs/proj-1/TASK-0001/}

## 输出要求
1. 将所有产出文件写入目录：{collaboration_dir}/{TASK-XXXX}/
2. 所有产出文件写完后，最后生成 `result.json`
3. 先写入 `result.json.tmp`，确认写完并 flush 后，再原子重命名为 `result.json`
4. `result.json` 至少包含：`task_code`、`summary`、`artifacts`
5. 完成后执行 git add、git commit、git push

{如果勾选了“输出剩余用量”：}
6. 在 {collaboration_dir}/{TASK-XXXX}/usage.json 中写入当前剩余用量信息
```

模板中的以下模块为 MVP 最小必需集合：

- 项目任务介绍（仅在项目 `goal` 非空时出现）
- 模版所需信息（仅在 task 可追溯到 `template:<template_id>` 且项目 `template_inputs` 有对应非空值时出现）
- 任务信息
- 前序任务输出路径
- 输出要求
- 可选用量输出要求

### 任务状态跟踪机制

每个 Prompt 都包含唯一任务码和固定任务输出目录。Agent 完成任务后，需将真实产物写入该目录，并在最后一步原子提交 `result.json`。系统轮询项目仓库并按以下规则判断状态：

1. 固定完成契约：任务目录固定为 `{collaboration_dir}/{task_code}/`；轮询只检测该目录下的 `result.json` 是否存在。系统不再依赖 `expected_output_path` 的自然语言内容来猜测真实产物路径。
2. 结果文件约束：`result.json` 是完成哨兵，不是中间过程文件；所有真实产物必须先写完，再写 `result.json.tmp`，最后原子重命名为 `result.json`。推荐其至少包含 `task_code`、`summary`、`artifacts`。
3. 可选用量文件：若任务上报用量，则写入同目录下的 `usage.json`。
4. 状态推进：检测到 `result.json` 后，将任务标记为“已完成”；超过超时时间仍未检测到结果，则进入“需人工处理”。
5. 恢复策略：`needs_attention` 状态任务不会停止检测；一旦后续检测到 `result.json`，系统应自动恢复其为“已完成”，并清除当前 `last_error`。
6. 手动完成：负责人通过“手动标记完成”接口完成任务时，系统也应清除当前 `last_error`，避免出现“已完成但仍保留旧错误”的脏状态。
7. `git fetch` / clone 等导致远端不可达的同步失败必须向任务或计划写入可读错误；网络抖动场景下系统必须有限次重试并指数退避。同步失败与“文件确实不存在”必须区分开，不能把同步异常吞掉后伪装成 `result not found`。若 `git fetch` 成功但 `git pull --ff-only` 因分叉失败，系统应将其作为可降级 warning 记录到后端日志中，不写入 `last_error`，并继续执行结果检测；此类 warning 不应屏蔽任务超时判定。

该机制属于弱一致性机制：系统依赖 Agent 遵循 Prompt 输出约定，不能独立验证任务质量，只能验证回写格式和文件存在性。

任务编辑与派发之间还需满足一个前端约束：若负责人刚修改了任务名称、描述或预期输出，系统必须先完成自动保存并基于最新服务端数据重新生成 Prompt，随后才允许点击“复制 Prompt 并派发 / 重新派发”。在新 Prompt 预取完成前，派发按钮应保持 disabled，避免旧 Prompt 与最新任务文本不一致。

### Agent 间信息共享机制

每个任务完成后，其输出文件保存在项目 Git 仓库的约定目录中。后续任务的 Prompt 自动包含前序任务输出文件路径，Agent 可直接从本地仓库读取前序结果，从而实现跨 Agent 的信息共享。

MVP 中仅考虑命令行类 Agent 的本地文件读取场景，不包含网页对话类 Agent 的上下文补录机制。每个任务的输出目录由唯一 Agent 写入，不同任务写入不同路径，天然避免 Git 并发写入冲突。

### 页面更新机制

MVP 采用 **可配置随机间隔轮询（全局默认 15-30 秒，支持项目级覆盖）+ 可配置延迟启动（全局默认 0 秒，支持项目级覆盖）+ 手动刷新按钮** 的方式更新页面状态，不承诺实时推送。真正的结果检测由后端按项目配置执行；前端页面不再固定间隔主动触发轮询，仅在进入页面、窗口重新聚焦/切回可见以及用户点击手动刷新时拉取最新状态。全局项目参数可在"项目参数设置"页面修改，包含轮询间隔、轮询启动延迟和默认 Task 超时时间；新建项目时页面会先读取当前全局默认值作为初始值，项目创建成功后将该组参数快照保存到项目，后续全局变更不追溯影响既有项目。

### 执行汇总

所有任务完成或终止后，项目详情页生成执行汇总页面，展示：

- 各任务当前状态
- 各任务产出文件链接
- 人工介入记录
- 最终项目结果摘要

MVP 的执行汇总以页面展示为主，不要求额外生成结构化报告文档。

## 端到端使用场景示例

1. **创建项目：** 项目负责人创建“多 Agent 文献综述与实验方案设计”项目，选择 2 个命令行类 Agent，并填写项目 Git 仓库地址。
2. **生成工作计划：** 负责人可选择由 Prompt 生成流程，也可选择已有流程模版并映射项目 Agent 后直接生成任务。
3. **计划调整：** 系统将两个计划渲染为可读 DAG，负责人合并、调整并确认最终执行方案。
4. **执行阶段：** 一个 Agent 负责文献检索与总结，一个 Agent 负责实验设计草案，任务依赖关系由页面控制。
5. **状态跟踪：** Agent 将结果回写到固定任务目录；系统通过 `{collaboration_dir}/{task_code}/result.json` 完成哨兵检测，将任务标记为“已完成”或“需人工处理”。
6. **异常处理：** 若某 Agent 超时未回写，负责人可选择重新派发、手动标记完成或标记放弃。
7. **执行汇总：** 所有任务结束后，系统展示任务状态、关键输出文件链接和人工介入记录。

## 异常与人工介入机制

系统定义以下 5 类任务状态：

| 状态 | 触发条件 | 界面展示 | 负责人可执行动作 |
|------|----------|----------|------------------|
| **待派发** | 任务已创建，但尚未点击复制 Prompt | 灰灯 | 复制 Prompt 并发送给 Agent |
| **执行中** | 已点击复制 Prompt，系统开始轮询，但尚未确认有效回写 | 红灯 | 等待、手动刷新 |
| **已完成** | 检测到有效输出产物；`.json` 结果需通过 `task_code` 校验，非 `.json` 结果需命中预期文件路径 | 绿灯 | 查看结果、继续后续任务 |
| **需人工处理** | 超时未回传 / 回写格式错误 / 输出不完整 / Git 读取失败 | 黄灯 + 原因提示 | 重新派发（重新生成 Prompt 并复制到剪贴板，由负责人粘贴给 Agent；上一次错误归档至事件日志）、手动标记完成、标记放弃 |
| **已放弃** | 负责人主动标记任务放弃 | 灰灯 + 删除线 | 查看记录；该状态为终态，不可重新开启，如需重新执行需创建新任务 |

**复制 Prompt 后的容错机制：** 若负责人点击复制 Prompt 后超过设定时间（默认 5 分钟）仍未检测到任何 Git 变更，系统提示“是否已将 Prompt 发送给 Agent？”供负责人确认或重新操作。

**任务超时机制：** 执行中任务超过设定超时时间后，自动转为“需人工处理”。默认 Task 超时时间由全局项目参数提供，取值范围 1-120 分钟，默认 10 分钟；新建项目时读取当前全局默认值并写入项目级默认，项目负责人可在项目创建/编辑页修改。最终计划生成 Task 时，将项目级默认超时时间快照写入每个 Task；Task 处于 `pending` 状态时可继续编辑该超时时间，进入 `running`、`needs_attention`、`completed`、`abandoned` 后不再允许修改。轮询判定统一使用有效超时时间，正常数据以 Task 自身值为准，防御性兜底顺序为 Task 值、项目级默认、全局默认、10 分钟。界面仅展示"超时时间"，不展示继承来源。

**并行任务部分完成时的处理：**

- 若后续任务仅依赖已完成部分，则“下一步”提示可继续开始
- 若后续任务依赖全部完成，则显示等待状态并标注尚未完成的任务

## 系统功能模块

当前 MVP 左侧导航固定提供”项目”、”智能体”和”流程模版”入口；管理员额外可见用户管理入口。不提供独立”修改记录”页面。

左侧导航底部展示当前登录用户的欢迎信息（”欢迎您，XXX”），位于”退出登录”按钮上方，用于明确当前操作身份。

### 登录页

提供简单密码认证入口。

### 项目创建页

用于创建/编辑项目，填写项目目标、选择参与 Agent、配置 Git 仓库地址和项目参数。规划模式不在该页面展示，改由 Plan 页面在 Prompt 路径下选择和保存。

### 流程模版页

用于管理可复用的任务流程。负责人可在新建/编辑模版页先填写模版名称和适用场景，再输入详细流程描述生成面向外部 Agent 的模版编写 Prompt；Prompt 生成时会把适用场景作为流程目标上下文、把流程描述作为详细需求，并要求外部 Agent 在 JSON 中输出 `agent_roles` 角色说明。“输入描述”区块在“生成 Prompt”右侧提供“拷贝 Prompt”按钮：未生成 Prompt 时禁用，生成后复制当前 Prompt 文本（包含用户在 Prompt textarea 中的手工编辑），复制失败时在页面顶部展示 `拷贝 Prompt 失败：...`；该动作只写入剪贴板，不调用派发、轮询或其他后端接口。详细流程描述保存为模版的 `prompt_source_text`，编辑态打开时自动回填；为空时输入框为空，详情页展示“暂无说明”。用户随后手工粘贴和调整模版 JSON，预览 DAG 后保存。新建和编辑两态均展示“1. 基本信息 / 2. 输入描述 / 3. 粘贴 JSON或编辑 JSON”的完整链路，并提供“必需输入信息”编辑区，用于增删、排序和校验 `required_inputs`；该编辑区的每一行必须使用稳定的前端本地行标识维持输入焦点和行状态，不得使用正在编辑的 `key` 字段作为列表渲染标识。预览时仅在页面名称或适用场景为空时，才从 JSON 的 `plan_name` / `description` 回填，不覆盖用户已填写内容；DAG 预览区必须有稳定高度，避免 React Flow 在容器高度未确定时空白。角色说明区在成功预览并抽取到槽位后展示，槽位数量由 JSON 中 `tasks[].assignee` 决定，JSON `agent_roles` 可同步未手工编辑的说明，但不得覆盖用户已手工编辑的 slot；用户可修改说明但不能在该区域独立增删 slot。列表页展示模版名称、说明、角色槽位数、创建者和更新时间；详情页展示详细描述、只读 JSON、DAG、每个 slot 的只读角色说明，以及已声明的必需输入字段；创建者和管理员可编辑或删除，其他用户只能查看和使用。

### 项目详情页

作为核心工作台，展示“下一步应该做什么”、任务总体状态、计划生成入口、执行入口与异常提示。

### 计划查看与调整模块

负责将 Agent 输出的结构化计划渲染为可读内容，并以 DAG 方式展示任务节点与依赖关系，支持负责人手工调整。

### 任务执行模块

负责按 DAG 顺序展示待执行任务，生成 Prompt，支持一键复制，并提供“输出剩余用量”的可选勾选框。

### 状态跟踪模块

负责轮询 Git 仓库、解析任务回写结果、校验任务码、推进任务状态，并在页面中展示结果。

### 执行汇总页面

用于展示项目结束后的任务状态、产出文件链接、人工介入记录与结果摘要。

### Agent 总览页面

集中展示所有已登记 Coding Agents 的状态信息。采用单列卡片式布局（每个 Agent 占据一行），每张卡片展示以下信息：

- Agent 名称 + 可用状态徽章（视觉主锚点）
- Agent 类型介绍（来自智能体设置页面，展示该类型的能力与使用限制说明）
- 类型和一个或多个模型名称（紧凑徽章形式）
- 订阅到期时间（徽章形式）
- 每个模型对应的能力描述（按模型分行展示，必要时截断）。去重规则：同一模型在多个 Agent 中出现时，仅在自动排序顺序下首次出现的 Agent 卡片中显示能力描述，后续出现时省略（手动排序不改变此判断基准）
- 短期/长期重置剩余时间（倒计时格式，悬浮显示具体时间；短期不足 1 小时显示红色，长期不足 1 天显示红色、不足 2 天显示橙黄色）
- 短期/长期重置间隔
- 当前可用状态：采用四状态模型（可用 / 不可用 / 短期重置后可用 / 长期重置后可用），具体规则如下：
  - **可用**：当前 Agent 可正常使用。用户点击状态徽章时出现下拉框，可手动切换为”短期重置后可用”或”长期重置后可用”。
  - **短期重置后可用**：用户选择后，系统自动读取下一次短期重置时间，状态显示为”X/X XX:XX可用”（如”4/6 11:22可用”）。当短期重置时间到达、或用户点击短期重置处的重置按钮时，状态自动变为”可用”。用户也可随时点击状态徽章手动变更。
  - **长期重置后可用**：逻辑与”短期重置后可用”相同，但读取的是下一次长期重置时间。
  - **不可用**：当当前时间晚于订阅到期时间时自动进入此状态，且此状态不可通过下拉框切换。只有更新订阅到期时间为未来时间后，才会自动恢复为”可用”。
  - **隔离原则**：状态切换操作仅修改 `availability_status` 字段，不得影响 Agent 的模型配置、能力描述、重置策略等其他属性。前端使用独立的 PATCH 接口完成状态变更，后端在该接口中不调用通用的字段归一化逻辑。
- 订阅到期时间不足 48 小时时，到期时间以红色字体醒目显示

卡片排序规则：
- **自动排序**（系统默认）：首先按状态分组，"可用"排最前，"短期重置后可用"和"长期重置后可用"排中间，"不可用"排最后。"可用"组内按"最近一次重置时间"排序——取短期重置时间和长期重置时间中较早的一个作为该 Agent 的最近一次重置时间，最近的排前面，未设置重置时间的排在最后；"短期/长期重置后可用"组内按对应重置时间从近到远排序。
- **手动排序**：用户可通过拖拽卡片调整顺序，拖拽结果持久化到 `display_order` 字段。手动排序后，页面头部"新增智能体"按钮左侧出现"自动排序"按钮（蓝色风格），悬浮提示排序规则说明；点击后恢复系统默认排序。若当前顺序已与自动排序一致，则该按钮不显示。

卡片视觉区分：当 Agent 状态为非"可用"（即"短期重置后可用"、"长期重置后可用"或"不可用"）时，卡片底色切换为淡灰色（#f8f9fb），名称和能力描述降低透明度，以在视觉上与可用 Agent 区分。

“同服务器”字段用于标记 Agent 是否默认与项目部署机器同服务器；项目创建/编辑页可对每个参与 Agent 覆盖该值，覆盖只影响当前项目，不修改 Agent 默认值。

当 Agent 状态为"不可用"且不属于当前项目原有关联集合时，项目创建/编辑页中的该 Agent 卡片仍需展示，但以禁用态呈现并提示"不可用，无法参与项目"；编辑态下已保留的不可用 Agent 不应被误显示为禁用，以避免用户误判历史绑定丢失。

Agent 信息由负责人在系统中手动登记和维护。每个 Agent 允许配置一个或多个模型，每个模型的能力描述统一在"智能体设置"页面维护（详见下文），新增/编辑 Agent 时能力描述字段为只读。短期重置时间、长期重置时间、短期重置间隔、长期重置间隔均为选填；负责人录入时可先选择原始时区（如 UTC / GMT / EST / EDT），系统自动换算为北京时间后保存与展示，精确到分钟。

编辑智能体时，系统需要支持**局部更新**：用户只修改单个字段（如 `capability`、`co_located` 或 `name`）时，未提交的模型相关字段（如 `model_name`、`models_json`）不得被清空。只有在显式提交 `models` 配置时，后端才重建主模型与能力摘要字段。

长期重置间隔支持两种模式：
- **按天**（默认）：用户输入天数，每隔 N 天重置一次。
- **每月**：每月同一天同一时间自动重置。重置日期和时间由"下次重置时间"决定，例如设为 5 月 2 日 13:17，则每月 2 日 13:17 重置。表单下拉动态显示"每月X日 HH:MM"。若当月无对应日期（如 31 日），则取当月最后一天。
为避免浏览器、后端和 SQLite 对时区的重复换算导致分钟偏移，短期重置时间和长期重置时间在系统内部统一按”北京时间本地时间”处理：前端提交的是换算后的北京时间年月日时分，后端落库时不再二次转 UTC，展示时也不再做二次时区推导。

新增/编辑智能体采用模态表单，按”基本信息”、”模型与能力”、”短期重置策略”、”长期重置策略”四个分区组织字段，每个分区使用独立的区块卡片。”模型与能力”分区支持动态增加多个”模型”配置行，模型的能力描述为只读字段，内容来源于智能体设置页面的模型定义。

项目创建/编辑页面的 Agent 选择从复选框行改为可点击的选择卡片，每张卡片展示 Agent 名称、类型、全部模型名称、能力概要和状态，选中时高亮边框和背景。这里的可用状态显示规则需与 Agent 总览页面一致，使用同一套四状态推导逻辑。规划模式不在项目创建/编辑页面展示；负责人进入 Plan 页面并选择“由 Prompt 生成流程”时再选择本轮规划模式，该模式只影响后续 Plan Prompt 的规划策略，不改变可选 Agent 列表或轮询配置。

### 项目参数设置页面

项目参数设置页（`/settings`）用于维护全局项目参数和规划 Prompt 引导参数，仅管理员可修改。页面包含轮询间隔、轮询启动延迟、默认 Task 超时时间，以及“Prompt 设置”分组。

“Prompt 设置”分组包含“同机分配引导”多行文本配置。系统未保存过配置时显示后端默认文案；管理员可保存非空自定义文案，也可点击“恢复默认值”将文本重置为后端默认文案。保存后的文案写入 `global_settings`，并在后续新生成或重新生成的规划 Prompt 中生效。该配置不允许为空，前端应阻止空白保存，后端也必须拒绝空白值。

### 智能体设置页面

Agent 总览页面标题栏”新增智能体”按钮旁设有”设置”按钮，点击后进入智能体设置页面（`/agents/settings`）。该页面用于管理 Agent 类型和模型的全局配置，仅管理员可见、可用；普通用户既看不到入口，也不能通过接口修改。该页面用于管理 Agent 类型和模型的全局配置：

**Agent 类型管理**

- 用户可增加、删除、修改 Agent 类型列表
- 每个 Agent 类型有一个 **Agent 介绍** 字段，用于描述该类型智能体的能力、使用限制等说明（例如：1M Context、速度更快、支持多模态输入等）。在项目执行任务和 task 时，系统会结合 Agent 介绍与模型能力描述，以及 task 内容，进行最优化匹配，选择最合适的模型来完成 task
- 同名 Agent 类型共享同一份 Agent 介绍（由 `agent_type_configs` 表唯一约束保证），一处修改所有引用处同步更新
- 新增或修改 Agent 类型时，用户每输入一个字符，系统从已有类型中自动推荐匹配项（搜索补全）
- 删除类型时，若该类型下已有已创建的 Agent，则禁止删除

**模型管理**

- 每个 Agent 类型下可配置一个或多个模型
- 新增或修改模型时，用户每输入一个字符，系统从全局已有模型中自动推荐匹配项（按名称和别名搜索）
- 每个模型有两个属性：
  - **别名**（选填）：该模型名称的别名
  - **能力描述**（≤150 字）：描述该模型擅长的能力、适合的任务场景、性价比等。在项目执行选择模型时，系统会重点参考此处设置的能力描述选用最合适的模型

**模型身份识别规则**

- 如果用户输入的模型名称与系统中已有的某模型的名称或别名一致，则识别为同一模型，共享能力描述（一处修改，另一处同步更新）
- 如果用户输入的模型别名与系统中已有的某模型的名称或别名一致，同样识别为同一模型，共享能力描述
- 新增 Agent 时，模型下拉列表来自智能体设置页面配置的模型，能力描述字段为只读

为避免把智能体设置接口收紧后误伤普通用户创建/编辑智能体，系统单独提供只读的类型目录能力给普通用户使用，用于表单中的 Agent 类型和模型选择；该只读目录不具备任何修改能力。

### 用户管理页面

管理员登录后，左侧导航栏新增”用户管理”入口（`/admin/users`）。页面提供系统用户总览与基础账号管理能力：

- 查看字段：
  - 用户名
  - 注册时间
  - 最后一次登录时间
  - 用户类型（管理员 / 普通用户）
  - 用户状态（正常 / 冻结中）
- 支持操作：
  - 将普通用户设为管理员
  - 将管理员设为普通用户
  - 冻结用户
  - 解冻用户

交互约束：

- 修改角色前必须二次确认
- 冻结用户前必须二次确认
- 解冻不需要确认
- 当前登录管理员自己的危险操作按钮必须禁用
- 最后一个激活管理员的降级 / 冻结按钮必须禁用

后端约束：

- 管理员不能修改自己的角色
- 管理员不能冻结自己
- 系统至少保留一个激活状态的管理员

操作审计：

- 用户角色变更
- 用户冻结 / 解冻
- 用户修改自己密码

上述操作都必须进入操作审计日志，至少记录操作者、动作类型、目标对象、目标 ID、时间以及必要的结构化 detail；detail 中不得包含密码明文或密码哈希。

管理员可通过审计日志接口查看最近的操作记录，并按 `action` 过滤。

### 用量信息获取机制

各 Agent 的精确用量无法稳定通过 Prompt 获取，因此 Agent 总览页不再展示“用量情况”与“用量更新时间”。改由负责人手动维护两个辅助时间字段：

- 短期重置时间：例如下一次小时级或短窗口额度恢复时间
- 短期重置间隔：例如每隔 5 小时自动续推一次短期重置时间
- 长期重置时间：例如下一次日级、周级或长窗口额度恢复时间
- 长期重置间隔：例如每隔 7 天自动续推一次长期重置时间

如果某个 Agent 同时设置了“重置时间”和对应“重置间隔”，当该时间到期后，系统会自动将该时间向后顺延一轮：

- 短期重置时间到期后，自动更新为“原时间 + 短期重置间隔（小时）”
- 长期重置时间到期后，自动更新为“原时间 + 长期重置间隔（天）”

若某个 Agent 同时设置了“重置时间”和对应“重置间隔”，并且系统发生了自动续推，则对应“重置剩余时间”单元格需要在倒计时下方出现两个按钮：

- `重置`：鼠标悬浮提示“点击重置短期剩余时间”或“点击重置长期剩余时间”，按钮使用黄色底色以强调人工校正操作。点击后，系统把对应重置时间改为“当前北京时间 + 对应间隔”，随后隐藏两个按钮。
- `确认`：鼠标悬浮提示“下次短期重置时间无误”或“下次长期重置时间无误”。点击后不修改时间，仅隐藏两个按钮。

如果某个 Agent 没有设置对应的重置时间或重置间隔，则不显示这两个按钮。
如果负责人在按钮显示期间，通过编辑手动修改了对应的重置时间或重置间隔，则这一行对应的按钮应立即消失。

**联动规则：** 点击长期重置的"重置"按钮时，如果该 Agent 同时设置了短期重置时间和短期重置间隔，则视为同时点击了短期重置的"重置"按钮（即短期重置时间也会被重置为"当前北京时间 + 短期间隔"，短期确认标记也会被清除）。反之，点击短期重置的"重置"按钮不影响长期重置。

如系统中存在旧版本按 UTC 误存的重置时间，升级后需在启动阶段一次性校正为北京时间，避免页面显示与用户输入不一致。

> 机制边界说明：这两个时间字段为人工维护的辅助参考信息，不保证实时与绝对准确。

## MVP 技术方案概要

MVP 推荐采用以下技术方案：

| 层 | 选型 | 理由 |
|----|------|------|
| 后端 | Python 3.12 + FastAPI | 开发速度快，生态成熟，便于处理 Git、轮询、结构化数据 |
| 数据库 | SQLite | 轻量、零运维，适合当前轻量多用户隔离的 MVP |
| ORM | SQLModel 或 SQLAlchemy | 与 FastAPI 集成良好 |
| 前端 | React 18 + Vite + TypeScript | 构建快，与后端职责分明，组件生态成熟 |
| 路由 | React Router | 轻量，适合 SPA |
| DAG 可视化 | React Flow | 开源、交互成熟，适合任务依赖展示与编辑 |
| Git 集成 | 优先 git CLI，必要时引入 GitPython | 满足 clone、pull、读取、轮询需求 |
| 部署 | Docker Compose | 适配当前服务器环境，部署成本低 |

系统架构采用三层划分：

- Web 前端负责页面展示、交互编辑与手工触发操作
- 后端服务负责项目管理、Prompt 生成、状态推进与 Git 轮询
- SQLite 存储系统状态，Git 仓库存储项目协同结果

## 核心合规性设计

- **全程人工分发：** 所有任务均由用户手动复制粘贴发送，无自动提交、无自动调用、无模拟操作。
- **状态通过 Git 轮询实现：** 不依赖非公开接口或逆向方式获取状态。
- **结果可追溯：** 所有产出归档至 Git，便于审计和回看。
- **明确边界：** 系统负责流程编排、状态跟踪与归档展示，不直接替用户操作外部 Agent。

## 非功能性需求

- **部署形态：** Web 应用，支持自托管部署。
- **用户规模预期：** MVP 以小规模团队和单租户自托管场景为主，不以大规模多人并发作为设计目标。
- **数据安全：** 任务数据不应包含凭据、密钥等敏感信息，Prompt 模板中应有相应提示。
- **可用性：** 轮询服务需具备自动重启能力，Web 服务提供基础健康检查。MVP 不做高可用架构。

## 版本边界

| 版本 | 包含功能 | 目标 |
|------|----------|------|
| **MVP** | 简单密码登录、项目创建、Git 仓库地址配置、工作计划生成（JSON）、结构化计划渲染与 DAG 调整、Prompt 分发与一键复制、固定任务目录 + `result.json` 哨兵状态跟踪、Agent 总览页面（可用状态/订阅到期/短期长期重置时间与间隔）、任务级可选用量上报与展示、执行汇总页面、基础异常处理 | 跑通真实 Agent 闭环演示 |

## 已确认决策

以下事项已在评审过程中全部定案：

1. **结构化计划格式**：统一 JSON
2. **任务完成契约**：每个任务固定输出到 `<collaboration_dir>/<任务码>/`；所有真实产物先写入该目录，最后以原子方式提交 `result.json` 作为完成哨兵。若上报用量，则写入同目录的 `usage.json`。`expected_output` 在当前版本主要承担展示/参考语义，不再作为轮询检测的定位依据。
3. **前端技术栈**：React 18 + Vite + TypeScript
4. **后端技术栈**：Python 3.12 + FastAPI + SQLite
5. **用量上报查看**：任务详情展示 + Agent 总览页面
6. **DAG 编辑交互**：React Flow 可视化展示 + 侧栏表单编辑，MVP 不做拖拽连线
7. **工作计划评审**：MVP 不做，归入 V1

详细实现规格（数据库 schema、API 接口清单、页面路由与交互说明）见 `tech_spec.md`。

## 非功能需求：页面加载体验

HALF 是一个高频使用的多 Agent 操作台，页面切换体感必须接近“立即可见”。具体要求：

- 首次加载主菜单中的轻页面（如项目列表）不应被任务页、Agent 页等重页面的体积拖慢，前端必须按路由拆包。
- 跨页面共享的低频数据（Agent 列表、项目基本信息）切回页面时应立即显示上次的内容，再在后台静默刷新，不允许整页 loading 阻塞；若后台刷新失败但本地仍有最近一次成功数据，则页面应继续保留旧内容而不是抛出前端异常。
- 任务页 / Plan 页的轮询、focus 刷新、可见性刷新不应叠加成短时间多轮请求；写操作完成后应做局部更新而不是整页重拉。
- 后端用于支撑列表 / DAG 的批量接口（如前序输出状态接口）不允许出现按任务数线性放大的重复查询或重复 git 操作。

实现细节与代码约定见 `docs/tech_spec.md` 第七节“前端加载性能约定”。
